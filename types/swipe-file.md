# Type: Swipe File / Static Curated Asset

Use when the offer's paid mechanism is **execution, infrastructure, or scaled delivery** — not diagnosis. This type fits when the prospect typically has little or no existing attempt to audit (a diagnostic quiz would be diagnosing a void), and when the gap is "they don't know how much is hidden behind doing this well," not "they're doing it wrong and don't know why."

No scoring, no branching, no logic. If the offer's value is accountability or correction rather than execution/infrastructure, this is probably the wrong type — check `diagnostic-quiz.md` instead.

This file is the **shape** of the checklist. Populate every bracketed part with specifics from Steps 1-3 before revealing to the user.

---

## The checklist shape (8 steps)

### 1. Select the examples
Pull [3-5] real examples (the creator's own work, or honest composites) that demonstrably worked — each needs a real outcome attached (a number, a result), not just "this looks good."

✅ Pass bar: every example has a stated, specific outcome — not vague quality claims.

### 2. Identify the teaching point per example
For each example, name the *one* specific decision that made it work — not a generic "why this is good" summary.

✅ Pass bar: each example teaches something different from the others. No redundant lessons across the set — if two examples teach the same thing, cut one.

### 3. Write the annotation for each decision
Explain the specific mechanism behind the decision (not "this works because it's personalized" — explain *why* this exact form of personalization, this exact timing, this exact structure).

✅ Pass bar: reads as the creator's actual reasoning, not a generic listicle that could've been written by anyone in the niche.

### 4. Expose the gap deliberately
Each annotation explains the *what* and *why* clearly enough to be genuinely useful — but stops short of the systemized *how to execute this reliably at scale* (the infrastructure, the repeatable process, the part that requires ongoing management).

✅ Pass bar: someone could execute one instance of this by hand using only the swipe file — but couldn't run it as a repeatable system without what's being deliberately left out.

### 5. Write the close
Name or imply the gap left open in Step 4 — the execution/infrastructure layer the paid offer actually handles. No direct pitch.

✅ Pass bar: no direct ask; the remaining gap is clear without being stated as a sales line.

### 6. Format as a static asset
PDF or structured doc (e.g. Notion). No logic or branching required. Format for scanability — one header per example, not a wall of text. Match the creator's existing brand/voice.

✅ Pass bar: each example is visually separated and scannable in under 30 seconds per section.

### 7. Capture mechanism
Decide gate placement (email/DM before delivery) based on the creator's existing funnel — don't introduce a new capture method if one already exists in their stack.

### 8. Build, connect, QC
Confirm delivery mechanism works end-to-end (link resolves, file downloads/displays correctly). Then run the full pass in `references/qc.md` — do not skip this because "it's just a PDF."

## Execution layer

**Recommended tools (ranked):**

1. **Notion (public page) or PDF export** — fastest, no design skill required, easy to update later without re-sending a file, clean default formatting. Best fit for most cases.
2. **Canva (designed PDF)** — more visually polished, worth it if the creator already has brand assets/templates in Canva and wants a more "designed" feel.
3. **Plain Google Doc → PDF export** — fine as a last resort if speed matters more than polish; least differentiated visually.

**Full build-out — Notion:**

1. Create a new Notion page, one section per example (use a Heading 2 per example, not a flat list).
2. Under each heading: paste the example itself (the actual sequence/copy/asset), then the annotation directly beneath it in a callout block — visually separates "the thing" from "the reasoning," which matters for scanability.
3. Add the close as a final section at the bottom — separated visually (divider line) so it doesn't read as part of the curated content itself.
4. Share → publish to web, or export as PDF depending on whether the creator wants it living as a link or a downloadable file.
5. If gating: place the capture step before the page link is revealed (e.g. DM automation delivers the link only after a tag/response, or a simple landing page with an email field gates the Notion link).
6. Connect capture to existing ESP/CRM/ManyChat per the creator's existing stack — same as any other lead magnet, no new tooling needed for this step.

**Production flags for this type:** no video/voice needed. Written assets needed: the examples themselves (curated, not written from scratch unless the creator lacks real examples), annotations per example, close line. Design need is light — consistent headers and spacing matter more than heavy visual design.
