# Type: Text-Based (Worksheet / Framework Doc / Template)

Use when the paid mechanism is **applying a framework correctly to one's own situation** — and the lead magnet's job is to hand over a structured tool the prospect fills in or follows themselves, rather than curated examples (use `swipe-file.md` for that) or something that must be seen/heard in motion (use `video-sequence.md` for that).

This differs from a swipe file specifically: a swipe file shows *proven external examples* with reasoning exposed. A text-based asset is an *original tool* — a worksheet, a fill-in-the-blank template, a structured framework doc — that the prospect actively uses on their own material, not just reads.

No scoring, no branching, no production. Lowest production cost of the four types — pure writing/structuring work.

This file is the **shape** of the checklist. Populate every bracketed part with specifics from Steps 1-3 before revealing to the user.

---

## The checklist shape (8 steps)

### 1. Confirm text-based is the right fit
Before building: does this need to be filled in/applied by the prospect (text-based fits), shown as proven external examples (swipe-file fits instead), or does it require audio/visual demonstration (video-sequence fits instead)? If the prospect has nothing of their own to apply the framework to, this type may be the wrong choice.

✅ Pass bar: a clear reason exists for why "hand them a tool to apply" beats "show them examples" or "show them in motion" for this specific offer.

### 2. Define the framework
Name the exact structure being handed over (a sequence of questions, a fill-in template, a decision framework). This should mirror a real piece of the creator's actual paid methodology — not a generic worksheet format.

✅ Pass bar: the framework is specific enough that two different creators in the same niche wouldn't independently produce the same one.

### 3. Build the structure
Lay out the actual fields/sections/prompts the prospect will fill in or work through, in order. Each prompt should do real diagnostic or clarifying work — not be filler to make the document look substantial.

✅ Pass bar: every field earns its place; removing any one would meaningfully weaken the output.

### 4. Write instructional copy per section
Brief guidance for each section — what to write, why this section matters, what a good answer looks like. Keep tight; this isn't an essay, it's instructions embedded in a tool.

✅ Pass bar: someone could complete the document correctly without needing to ask the creator a clarifying question.

### 5. Expose the gap deliberately
The framework should produce a genuinely useful output when completed — but stop short of telling the prospect what to *do* with what they find, or how to execute the next step at scale/with accuracy. The application/execution is what the paid offer handles.

✅ Pass bar: completing the document produces real insight, but the prospect is left needing help acting on it correctly.

### 6. Write the close
Name or imply the gap from Step 5 — what happens after they've filled this out, that the free asset doesn't cover. No direct pitch.

✅ Pass bar: implies the next problem without a direct sales line.

### 7. Format as a static asset
PDF, Google Doc template, or Notion template — pick based on whether the prospect needs to type directly into it (Doc/Notion duplicate) or just read and write elsewhere (PDF). Format for clarity: clear section breaks, enough white space to actually write in if applicable.

### 8. Build, deliver, QC
Confirm the delivery format actually works as intended (e.g. a "make a copy" Google Doc link works, a fillable PDF's fields function). Then run the full pass in `references/qc.md`.

## Execution layer

**Recommended tools (ranked):**

1. **Google Docs (template, "make a copy" link)** — free, prospect can fill it in directly without needing new software, easiest to update later. Best fit when the prospect needs to actively write in it.
2. **Notion (duplicate-able template)** — slightly more polished/structured feel, works well if the creator already delivers other things via Notion (keeps experience consistent).
3. **Static PDF** — simplest to produce, but weaker if the prospect is meant to write directly into it (PDFs are clunkier to fill in than a doc/Notion copy) — best reserved for read-only frameworks, not fill-in worksheets.

**Full build-out — Google Docs:**

1. Build the document with the structure from Step 3, instructional copy from Step 4 embedded in light gray/italic placeholder text per section.
2. Set sharing to "Anyone with the link can view" — not edit, to protect the original template from being altered by users.
3. Test the "File → Make a copy" flow works correctly so each prospect gets their own editable version, not the shared original.
4. Place the capture gate before the document link is revealed, matching existing funnel capture pattern.
5. Connect capture to existing ESP/CRM/ManyChat per the creator's stack.
6. Test end-to-end as a new lead would: capture → receive link → make a copy → confirm it's editable in their own account.

**Production flags for this type:** no video/audio needed. Written assets needed: framework structure, instructional copy per section, close line. Design need is minimal — clean formatting and clear section breaks matter more than visual design.
