# Type: Video Sequence

Use when the teaching point genuinely cannot be conveyed in static text — tone, pacing, timing, body language, or "the exact moment something went wrong" are part of what's being taught. If the same lesson could be written instead and lose nothing, this is the wrong type — use `swipe-file.md` or `text-based.md` instead. Video should be a deliberate choice because the medium carries information text can't, not a default for "feels more premium."

No scoring, no branching. Production-heavy — this is the main cost difference from other types.

This file is the **shape** of the checklist, grounded in a worked example (Closer's Edge — sales call coaching, 3-part call-breakdown sequence). Populate every bracketed part with specifics from Steps 1-3 before revealing to the user.

---

## Worked example reference

**Offer:** 1:1 sales call review coaching. Paid mechanism: catching the prospect's specific failure pattern in their own calls.
**Lead magnet:** 3-part video sequence, each video breaking down one real (anonymized) call clip showing one specific, isolated failure pattern.
**Why video fit:** the failure pattern is tonal/timing-based — "agreeing too fast," "explaining instead of asking" — and has to be heard in context to land, not described.

---

## The checklist shape (9 steps — one more than static types, to account for production)

### 1. Confirm video is the right medium
Before anything else: could this lesson be fully taught in text without losing anything? If yes, redirect to a static type. Video is justified when tone, timing, visual demonstration, or "watching it happen" is part of the teaching, not just a delivery preference.

✅ Pass bar: a clear, specific reason exists for why text/static couldn't teach this as well.

### 2. Select the source material
Identify [2-4] real examples to break down (real client work, anonymized calls/sessions, or a live demonstration the creator records themselves). Each needs a clear, specific teaching point — same standard as swipe-file Step 1.

✅ Pass bar: every example has a stated specific lesson, not "this is generally a good example."

### 3. Identify the teaching point per video
One isolated lesson per video — not a broad topic. ("This video is about agreeing too fast on price objections," not "this video is about objection handling.")

✅ Pass bar: each video's lesson is narrow enough to state in one sentence, and distinct from the others in the sequence.

### 4. Script the breakdown
Write the actual talking points per video — what gets said before showing the example, what gets pointed out during/after, how the lesson is named explicitly so the viewer can't miss it. This is not a teleprompter script necessarily, but the structure must be locked before recording — winging it on camera produces unfocused video.

✅ Pass bar: someone other than the creator could read the script and understand exactly what point is being made and where.

### 5. Expose the gap deliberately
Each video explains the *what went wrong* and *why* clearly — but stops short of teaching the prospect to reliably self-diagnose this in real time, in their own calls, without outside review. That ongoing blind-spot-catching is the paid mechanism.

✅ Pass bar: someone could understand the specific mistake shown — but couldn't yet catch the same mistake happening in their own next call without help.

### 6. Write the close (per video or final video only)
Name or imply the remaining gap — that this was one example, and their own pattern is likely different/hidden from them specifically. No direct pitch.

✅ Pass bar: implies "this is what *I* caught in this example — what's *your* pattern" without a direct sales line.

### 7. Record
Execute the script from Step 4. Keep each video tightly scoped to its one lesson — resist the urge to over-explain or add tangents; this directly violates the speed-to-value standard.

✅ Pass bar: each video stays on its single stated lesson start to finish.

### 8. Format and sequence
Decide delivery order (usually escalating specificity or building toward the close). Format consistently across all videos (same intro style, same call-out format) so the sequence feels designed, not assembled.

### 9. Build, deliver, QC
Confirm hosting/delivery works end-to-end (see Execution Layer). Confirm capture mechanism fires correctly. Then run the full pass in `references/qc.md` — apply the Production check section specifically (length/format/recording confirmed, not just scripted).

## Execution layer

**Recommended tools (ranked):**

1. **YouTube (unlisted) + simple landing/delivery page** — free hosting, reliable playback, easy to update/replace a video without rebuilding delivery. Matches the existing pattern already used for VSL hosting. Best fit for most cases.
2. **Loom** — fastest to record and share, built-in simple analytics (watch time), less polished as a "produced" asset — fine for a single breakdown video, weaker for a multi-part sequence meant to feel premium.
3. **Hosted course platform (e.g. existing delivery tool already in the stack)** — most controlled experience (sequential unlock, no skipping ahead), more setup overhead — only worth it if the creator already has this infrastructure for other delivery.

**Full build-out — YouTube (unlisted) + delivery page:**

1. Record each video per the locked script from Step 4.
2. Edit minimally — cut dead air, keep production light; polish matters less than clarity for a lead magnet (over-produced free content can undercut the paid offer's perceived premium quality).
3. Upload each video as **Unlisted** (not Public, not Private) — accessible via link, not searchable/discoverable.
4. Build a simple sequential delivery page (or use the existing landing page pattern from the funnel) listing the videos in order with one line of context per video.
5. Place the capture gate before the page/links are revealed — matches existing funnel capture pattern (email or DM-triggered link delivery).
6. Connect capture to existing ESP/CRM/ManyChat — same integration pattern as other types, no new tooling required.
7. Test the full path end-to-end as a new lead would experience it — capture → delivery → playback — before considering it built.

**Production flags for this type:** video/audio recording required — this is the defining cost of this type. Script required per video (locked before recording, per Step 4). Minimal editing needed, not full production value — the lesson clarity matters more than polish. No special design assets needed beyond a simple delivery page matching existing brand styling.
